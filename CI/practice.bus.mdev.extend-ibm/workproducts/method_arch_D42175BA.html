Artifact: Method Architecture
This artifact describes the architecturally significant elements of the method library and their relationships, as well as key information such as rationale, constraints, and key decisions.
Domains: Software Capability Improvement
Purpose
Analyze major architectural concerns and document architectural decisions. Assess and communicate the method architecture to stakeholders to ensure the architecture will support the method needs of the organization.
Relationships
RolesResponsible:
  • Method Developer
Modified By:

TasksInput To:

  • Implement Method Asset
Output From:
  • Architect the Method Library
Description
Brief Outline

The following are key pieces of information that should be described in this artifact:

  • Architecturally-significant decisions and rationale that drive content and organization of the elements in the library:
    • Plug-in types, what they should contain and their allowable dependencies
    • Configuration types and what should they contain
    • Naming conventions
    • Approach for defining navigation views
    • Approach for categorizing elements
    • Approach for assigning roles
    • Approach for scaling elements
    • Approach for associating guidance
  • Architecturally-significant structural elements of the library and their relationships:
    • Core method elements (for example: common standard categories, custom categories, roles, work products, and tool mentors)
    • Practices and their dependencies
    • Practice configurations
    • Architectural partitions, layers and their dependencies
    • Key method elements (for example: roles, tasks, work products. guidance, and processes)
Tailoring
Representation Options

The format of the artifact is not as critical as the content. Use whatever format would enable you to communicate with the stakeholders so that they can understand the architecture and provide feedback. However, when selecting a format, it may be helpful to consider the downstream activities that may be able to leverage these materials (for example, enablement) and choose a format that promotes such reuse.
