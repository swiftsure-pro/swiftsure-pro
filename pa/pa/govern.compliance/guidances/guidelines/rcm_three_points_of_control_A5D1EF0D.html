Guideline: Three Points of Control
Establishing a controlled software build and delivery workflow is key to a compliance-aware development process. The Three Points of Control guideline defines three key events where Control Points allow or prohibit continuation. This guideline introduces these controls.

Main Description

The Three Points of Control approach defines a controlled process for producing builds and delivering software. This approach builds in quality gates, defines clear lines of responsibility, and enforces a separation of duties by clearly delineating development, build management, and deployment activities.

1. The Change Control Point(or "Deliver Change") manages the incorporation of individual changes associated with one or more  Work Orders into the integration environment. During this control point, the Developer posts activities for approval by the Technical Lead. The Technical Lead can approve the change (allowing the incorporation) or disapprove it (sending the changes back to the  Implementer for rework). Once the change has been approved, no additional changes can be made to the change set associated with the work order(s).

2. The Baseline Control Point (or "Register Derived Objects") manages the addition of changes to source control. This control point occurs once development-level integration testing is complete, and captures the build results targeted for QA testing. Only the  Integrator has the authority to register build objects for QA.

3. The Release Control Point (or "Deploy Objects") manages the deployment releases to QA or Production environments. Once a registered release is ready for QA or Production deployment, this control point verifies that the work included in the release is complete, and that the build was performed in a controlled environment. Only the Release Manager has the authority to approve a deployment to QA or Production.
