Task: Validate the development process
This task describes how to execute the appropriate collections of tests that are required to evaluate the quality of the development process, as well as how to capture test results that facilitate ongoing assessment of the process.
Disciplines: rup_test_discipline
Purpose
The purpose of this task is to define how to run validation tests.

      Main Description

      The process validation objectives are to provide "reasonable assurance" of the correct implementation and operation of the process. This validates that the process works and satisfies applicable compliance requirements and policies. The tests performed should include:

      • Reports on metrics and the status of process use
      • Validation tests of automated tools that are used to implement the process
      • Verification that compliance goals are met
      Steps
      Set up your test environment for a Process Validation test
      Purpose:  To accurately establish the test environment in preparation for execution.

      Set up the test environment to ensure that all of the required components (hardware, software, tools, data, and so on) have been established, and that they are available and ready in the test environment in the correct state to enable the tests to be conducted. Although some tasks can be performed automatically, some aspects typically require human attention.

      Schedule Process Validation test execution
      Purpose:  To determine the appropriate time for test execution to begin.

      In many cases where test execution can be attended, the test suite can be executed relatively on demand. In these cases, scheduling will likely need to take into account considerations such as the work of other testers, other team members, and different test teams that share the test environment. In these cases, test execution will typically need to work around infrequent environment resets.

      It may be appropriate to use manual testing techniques in the compliance environment. The personnel doing the auditing are primarily accountants and quality personnel. These people are not engineers, so they will use techniques that they trust, such as manual testing and checking boxes. For example, Rule 404 of the Sarbanes-Oxley Act, in the United States, requires companies to perform internal reviews of their control systems, and then to hire an outside auditor to verify the review's findings. Testing the process and storing the results of the validation test would save time for the auditor and the internal accountants. This, combined with the metrics, provides a proof of use for the process.

      Run these tests quarterly, as required for the testing process. Update the versions of metrics reports monthly.

      Execute the Process Validation test suite
      Purpose:  To conduct the tests in the test suite and to monitor their completion.

      Executing the test suite will vary, depending on whether testing is conducted automatically or manually. In either case, the test suites developed during the test implementation tasks are used to either execute the tests automatically or to guide the manual execution of the tests.


      Evaluate execution of the Process Validation test suite
      Purpose:  To determine whether the test suite executed to completion or halted abnormally, and to assess whether corrective action is required.

      The execution of testing ends or terminates in one of two conditions:

      • Normal: All of the tests execute to completion as intended. 
      • Abnormal or premature: The tests did not execute completely as intended. When testing ends abnormally, the test logs from which subsequent test results are derived may be unreliable. The cause of the abnormal termination needs to be identified and, if necessary, the fault corrected, and the tests rerun.

      Inspect the test logs for completeness and accuracy
      Purpose:  To determine if the test suite execution generated worthwhile test information and, if not, to identify appropriate corrective action.

      When tests initially complete, review the test logs to ensure that the logs are reliable and that reported failures, warnings, or unexpected results were not caused by external influences (on the target of the test), such as improper environment setup or invalid input data for the test.

      If the reported failures are due to errors identified in the test work products or due to problems with the test environment, take the appropriate corrective action and rerun the tests.

      If the test log enables the Tester to determine that the failures are due to genuine failures in the test's target items, then the execution portion of the task is complete.


      Maintain traceability relationships
      Purpose:  To enable impact analysis and assessment reporting to be performed on the traced items. 

      Using the traceability requirements outlined in the Test Plan, update the traceability relationships as required. A good starting point is to consider traceability in terms of measuring the extent of testing or test coverage. As a general rule, the measurement of the extent of testing should be based on the motivators discovered during the test planning activities.

      Test suites might be traced to the defined test cases that they realize. They may also be traced to elements of the requirements, software specifications, design, or implementation.

      Regardless of what relationships have been selected as important to trace, the status of the relationships that were established during implementation of the test suite will need to be updated.

      Note: Traceability is a key aspect of compliance


      Evaluate and verify your results
      Purpose:  To verify that the task has been completed appropriately and that the resulting work products are acceptable.

      Now that the work is complete, it is beneficial to verify that the work was of sufficient value. Evaluate whether what was accomplished is of appropriate quality and that it is complete enough to be useful to those team members who will make subsequent use of it as input to their work. Where possible, use the checklists provided in the RUP methodology to verify that quality and completeness are adequate.

      At the conclusion of the test, the Tester attaches the test log to a permanent record of the test.

      Key Considerations
      This task executes tests that will validate adherence to audit requirements. The people doing the auditing are primarily accountants and quality assurance personnel. These people are not engineers, so they will use techniques that they trust. This is typically manual testing and checking boxes. Take this into account when defining these tests.
