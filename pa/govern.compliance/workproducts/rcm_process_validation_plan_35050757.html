Description
Main Description

Process validation objectives for compliance are to provide "reasonable assurance" of the correct implementation and operation of the software development process. This plan is, therefore, very similar to a test plan, but it focuses on validating that the process works, and it does not address the system being developed. It defines a set of tests that can be used to validate the software development process for compliance needs, including validated control points, audit trails, and security. Regulated environments that have similar rules and have been regulated for a long time have the following typical requirements:

  • Reports on metrics and the status of process use
  • Justification that the process effectively addresses relevant regulations, standards, and policies
  • Tests to demonstrate that process automation implements the process correctly

Because a process is not a completely automated mechanism, verifying that it fulfills its purpose requires developing a rationale that what it provides meets its base requirements. This includes:

  • Verifying that all of the claims of the compliance rationale are met by the process
  • Developing a narrative that explains how process features combine to satisfy the requirements of source regulations, standards, and policies.
  • Developing arguments that explains how process improvements will solve observed insufficiencies or observed nonconformance to target regulations, standards, or policies.

The focus and extent of the plan will change, depending on the primary goal of the project:

  • In a process modeling project, the plan covers all phases of the software development lifecycle and should assess the process as a whole, thereby establishing that the process fulfills its purpose.
  • In a software development project, the plan assumes the fitness of the overall process and focuses on validating that:
    •  The customizations or simplifications specified in the development case do not affect the effectiveness of the process
    • The process and its automation is in working order

Process testing is concentrated in the Inception phase, where the environment for a software project is defined.
