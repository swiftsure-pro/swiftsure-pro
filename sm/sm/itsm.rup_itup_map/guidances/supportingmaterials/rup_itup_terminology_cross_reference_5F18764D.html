Discipline
 A collection of related  tasks that define a major 'area of concern'. In software engineering, Disciplines include: Business Modeling, Requirements, Analysis & Design, Implementation,Test, Deployment, Configuration & Change Management, Project Management, and Environment. 		Not Applicable. Closest element in ITUP to a RUP discipline is a Process (see below).
Not Applicable. Closest element in RUP to an ITUP Process Category is a  capability pattern. Process Category
Grouping of related processes. Process categories express and communicate process knowledge for a key area of interest, such as IT Customer Relationships, and can be directly used by the practitioners to guide their work.

Process
 A general structure for particular types of development projects. Processes take content elements and relate them into semi-ordered sequences that are customized to specific types of projects. Thus, a Process is a set of partially ordered work descriptions intended to reach a higher development goal, such as the release of a specific software. These work descriptions are organized into a hierarchical breakdown-structure. A Process focuses on the lifecycle and the sequencing of work in breakdown structures.

Process
 A process is a related set of repeatable activities focused on achieving a set of common objectives in a measurable way. A process is defined by a set of activities, tasks, roles, and work products. A set of processes make up a process category. For example the ITUP processes for Solution Development process category are: Solution Requirements, Solution Analysis and Design, Solution Build, Solution Test, and Solution Acceptance. 

Activity
 A work  breakdown element which supports the nesting and logical grouping of related process elements such as  descriptor and sub-Activities, thus forming work  breakdown structures. 		Activity
 An activity is a segment of a process workflow. Processes are decomposed into activities. Each activity has inputs and outputs defined as work products.
Change
 Not Applicable. Closest element in RUP to a change is a request. See definition of RUP  change request. 		Change
 An action that results in a new status of one or more Configuration Item (CI), or the process of moving from one defined state to another, or the addition, modification or removal of approved, supported or baselined hardware, network, software, application, environment, system, desktop build or associated documentation.

Change Request
 A general term for any request from a stakeholder to change an artifact or process. Documented in the Change Request is information on the origin and impact of the current problem, the proposed solution, and its cost.

Request for Change
 A document (usually electronic) that details a requested change to a Configuration Item (CI). 

Change Control Board (CCB)
 The role of the CCB is to provide a central control mechanism to ensure that every  change request is properly considered, authorized and coordinated.

Change Advisory Board (CAB)
 The Change Advisory Board (CAB) assesses all high impact Request For Changes (RFCs) advises Change Management concerning the priorities of RFCs, and proposes resources to implement a change. The CAB may also participate in post-deployment reviews of changes.

The Change Management Framework identifies who approves an RFC. Typically, this is the Change Manager, with the CAB acting in an advisory capacity. Or the CAB may collectively approve the RFC, acting in both as advisors as well as approvers.

The CAB includes customer, user, and IT representatives from various functions, and is chaired by the Change Manager.

Configuration
 The configuration environment being managed includes the infrastructure of the software/system under development. Thus, “configuration” in RUP means the requirements, design, and implementation artifacts that define a particular version of a system or system component. 		Configuration
 The configuration environment being managed includes any systems within the entire IT infrastructure that are managed.  Thus, “configuration” in ITUP means any collection of desktops, laptops, network devices, software, printers, and other items that are managed within an enterprise.

Release
 A subset of the end-product that is the object of evaluation at a major milestone. A release is a stable, executable version of product, together with any artifacts necessary to use this release, such as release notes or installation instructions. A release can be internal or external. An internal release is used only by the development organization, as part of a milestone, or for a demonstration to users or customers. An external release (or delivery) is delivered to users. A release is not necessarily a complete product, but can just be one step along the way, with its usefulness measured only from an engineering perspective. Releases act as a forcing function that drives the development team to get closure at regular intervals, avoiding the "90% done, 90% remaining" syndrome.

Release
 A release is a collection of one or more approved changes that are deployed into the infrastructure together.

Service (in SOA context)
 A service is a discoverable software resource with an externalized service specification. This service specification is available for searching, binding, and invocation by a service consumer. The service provider realizes the service specification implementation and also delivers the quality of service requirements to the service consumer. Services shall be governed by declarative policies and thus support a dynamically re-configurable architectural style.

Service
 An IT capability that meets the needs of one or more IT customers. A service is what the customer experiences, whereas a solution is the set of resources that work together to provide a service.
