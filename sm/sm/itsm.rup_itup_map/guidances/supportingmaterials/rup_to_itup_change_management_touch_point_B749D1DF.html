RUP to ITUP Change Management Touch Point
This page provides an inside look at unifying the RUP Change and Configuration Management discipline with the ITUP Change Management process offering a heightened awareness of how software development processes affect and are affected by IT operations.
Main Description

The goal of this touch point is to take into account and prioritize any modification to or addition of a Configuration Item (CI) as requirements progress through application design and development.

When defining your change control process as part of performing the  Configuration & Change Management discipline, it is important that the task  Establish Change Control Process and specifically its step Establish the Change Request Process fully considers and addresses the interdependency between the Configuration & Change Management discipline and the Requirements discipline.

An example of this is when identifying and prioritizing requirements, it is important that the RUP  Requirements discipline takes into account all Requests for Change (RFC), their associated Known Error(s) and especially their priority (impact and urgency) as part of planning for a release. When applying requirements management using use cases, this information needs to be taken into account when performing the tasks  Prioritize Use Cases,  Find Actors and Use Cases,  Detail a Use Case and  Develop Supplementary Specifications. When applying requirements management using a more traditional approach, this information needs to be taken into account when performing the task  Detail the Software Requirements. As with other sources of requirements RFCs are used to clarify scope and direction with the project sponsor.

Another example is when verifying that an RFC has been completed. As part of performing the  Configuration & Change Management discipline and the task  Verify Changes in Build,  the development team needs to focus on ensuring that the build and subsequent testing continues to correctly address the agreed upon set of requirements. Since the development effort and the resulting build is based on modifications to the requirements from which the current application was derived or from new requirements, the underlying baselined requirements need to be managed effectively.  

It is worth noting that not all RFCs are for changes to software. For example:

  • IT operational requests for change may be specific to upgrading hardware such as workstations, printers or routers.
  • Once a new or modified application had been tested and is ready for deployment, one or more request for changes will be created to deploy the application in the live production environment.

As a summary, here are the relationships between RUP and ITUP elements at touch point:

RUP Elements ITUP Elements Relationship

Discipline:  Configuration & Change Management

  • Task:  Establish Change Control Process; specifically its step Establish the Change Request Process
  • Task:  Verify Changes in Build

Discipline:  Requirements

  • Task:  Prioritize Use Cases
  • Task:  Find Actors and Use Cases
  • Task:  Detail a Use Case
  • Task:  Detail the Software Requirements

ITUP Change Management Process Overview

  • Requests for Change (RFC)
  • Known Error
  • Configuration Item (CI)
