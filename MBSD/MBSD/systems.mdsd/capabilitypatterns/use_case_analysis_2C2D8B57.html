Steps
Create blackbox sequence diagrams

If we have completed our work through the previous MDSD step, what we have now is a complete set of enterprise use cases. The next step is to answer the question, “what operations must the enterprise be capable of, in order to make possible all of the usages described in these use cases?” To answer this question, we perform operation identification. Operation identification involves the use of a sequence diagram. Sequence diagrams show the same flow of events described in a use case, but use a very specific format and method to show them. The flow of events of each use case is shown as a series of interactions, more specifically requests from one system element to another. The use case is carried out as elements makes requests of one another.

 

Sequence diagrams in MDSD come in two flavors—black box and white box. The difference is in which system elements are used as the collaborating elements and placed at the top of the sequence diagram’s lifelines. In a black box sequence diagram, only the system and its actors are placed on the diagram while in a white box sequence diagram, multiple system elements are used in addition to the actors. For operation identification, we need only the black box sequence diagram.

For each enterprise use case, draw a sequence diagram with lifelines for the enterprise and each of the actors involved in that use case, or any of its alternate flows. Then, following the flow of events in the use case, write a sequence of requests that fulfill the use case. For example, consider the use case “commute to work” mentioned above. The enterprise is the car. The flow of events might begin:

 

  1. This use case begins when the driver approaches the car.
  2. The driver unlocks the car.
  3. The driver starts the car and allows it to warm up.
  4. The driver drives the car to the work location.
  5. etc.

We must “translate” this plain language flow of events into a series of requests. We do this by asking, for each step or set of steps in the use case, what request is being made of what element in the system do to something. Sometimes this takes a combination of imagination and reading ahead in the use case to determine the actual purpose of things.

 
Identify Operations

Once we have developed the black box sequence diagrams for each of the enterprise use cases, we are ready to identify the enterprise operations—our reason for doing all the sequence diagrams. Looking at a black box sequence diagrams, focusing on the enterprise element and you’ll see that some of the arrows are pointing in toward the element’s lifeline (the vertical line dropping from the element at the top of the diagram) and some arrows point away from this lifeline.

 

To determine the enterprise operations, note the arrows pointing into the lifeline only. When working with paper models, such as those on flip charts, we often circle these arrowheads in a bright color to emphasize their importance. Each arrow pointing in toward the enterprise element’s lifeline represents a candidate operation for the enterprise. Why? An arrow represents a message that carries a request—a request being made of the enterprise. If the system is to work, the enterprise must be capable of responding to that request when it is initiated. Thus the enterprise must have an operation that corresponds to the request.

 

To put it simply, if the enterprise is at some point requested, by any actor, to unlock, then it is required that the enterprise have an operation called “unlock.” It’s as simple as that. So, we can read the operations for the enterprise right off of the sequence diagram we have just drawn, by simply noting the arrows that point in towards its lifeline.
Re-factor and consolidate enterprise operations

It may seem that we have determined all of the operations necessary for the enterprise to fulfill all of its use cases, but there’s one final step. In most situations, we find that due to the elapsed time it takes to create complete enterprise use case models, and the fact that usually multiple modelers are involved, we must ensure that the operations determined from the analysis of the entire collection of use cases do not include redundant or overlapping operations.

 

To do this, review the list of operations for the enterprise that you have identified from analysis of all the enterprise use cases. Look for any operations that may be similar but named slightly differently. For example, if in one use case an operation was identified called “start-up” and in another “initialize” you might look more closely into these to see if they could be treated as the same operation. If so, then rename one or both of them so they are the same, and make any necessary adjustments top the use case flows of events and black box sequence diagrams to make it all consistent.

 

In the early stages of an MDSD model, you can expect lots of this kind of re-factoring and re-thinking of the model.
