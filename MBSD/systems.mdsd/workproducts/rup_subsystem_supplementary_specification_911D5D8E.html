Artifact: Subsystem Supplementary Requirements
This is essentially the Supplementary Specifications artifact one level down. It has to be a separate artifact because it has a different responsible role (at this level). Otherwise, it has the same purpose and content: it becomes the Supplementary Specifications for the application of RUP to the development of the subsystem.
Domains: Systems Engineering Work Products
Work Product Kinds: specification
Purpose

The following roles and role sets use the Supplementary Specifications:

  •  Analysts create and maintain the Supplementary Specifications, which serve as a communication medium between the analyst, the customer, and developers.
  • For software development:
    •  Developers use the Supplementary Specifications as a reference when defining responsibilities, operations, and attributes on classes, and when adjusting classes to the implementation environment.
    •  Implementers refer to the Supplementary Specifications for input when implementing classes.
  • In systems engineering:
    • The System Architect, in evolving the System Analysis, System Design and System Deployment models.
    • The System Designer, in evolving the System Use-Case Realization, Subsystem Use-Case Model and Subsystem Supplementary Requirements.
  •  Managers refer to the Supplementary Specifications for input when planning iterations.
  •  Testers use the Supplementary Specifications to verify system compliance.

Tailoring
Representation OptionsUML Representation:



The kinds of supplementary requirements vary widely between projects, so tailoring needs to be applied to define sections applicable to your project.  Decide which information (attributes) to include in the Supplementary Specifications, and which to manage using requirements management tools, such as Rational RequisitePro (see  Tool Mentor: Managing Dependencies using Rational RequisitePro®, and also see  Work Product: Requirements Attributes).
